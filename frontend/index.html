      🚀 Quick Start --- Read This First

      The tracker manages preconstruction bid tasks across multiple projects. Here's the absolute minimum to be useful:

      1
      Create a project. Click + New Project at the top of the sidebar. Enter the project name, bid start date, and bid due date. These two dates are the most important --- every task offset and milestone is calculated from them.
      2
      Load the master template (optional). When creating a project, you can load SBG's master task workflow --- it pre-populates dozens of standard tasks across all bid stages with day offsets calibrated from real experience. Saves 30+ minutes per project.
      3
      Assign tasks. Click any task to open its details. Set the Assignee (Lead Owner) and optionally a Support member. The Assignee is responsible for getting it done; Support is a helper.
      4
      Update status as you work. Drag cards between columns on the Board view, or use the inline status dropdown on each card. Statuses are: Not Started → In Progress → Blocked → Pending → Complete.
      5
      Check your alerts daily. The Alerts section at the top of any project shows what's overdue, due today, and what new tasks need acknowledgment. This is your morning check-in.

      💡 Identify yourself first: Open Settings → set your Current User name. This is how the system knows who's acknowledging tasks, signing off on completions, and posting messages. Without this, the audit trail breaks.

      🔍 Find anything fast: Click the search bar at the top of the sidebar (or press /) to search across every project, task, message, addendum, and team member at once. Type 2+ characters → results appear instantly grouped by type. Use ↑↓ to navigate, Enter to jump, Esc to close.

      🔍 Global Search --- Find Anything Fast

      The search bar at the top of the sidebar searches across everything in your tracker --- every project, task, message, addendum, scope package, and team member. One search box, complete coverage.

      Keyboard Shortcuts

      • / --- focus the search bar from anywhere (when not typing in another field)
      • Cmd/Ctrl + K --- focus the search bar (works even mid-edit)
      • ↑ ↓ --- move selection up/down through results
      • Enter --- jump to the highlighted result
      • Esc --- clear and close the search dropdown

      What's Searchable

      • Projects --- by name, owner, location, architect, or type
      • Tasks --- by title, notes, category, assignee, or support members
      • Messages --- both project-level and task-level message text
      • Addendum log --- by addendum number, title, summary, or notes
      • Scope packages --- by name or description
      • Team members --- by name, title, or email --- clicking jumps to that person's tasks in the active project

      How Results Work

      • Results appear as you type (after 2+ characters) --- no Enter required
      • Grouped by type with section headers (Projects, Tasks, Messages, etc.)
      • Each result shows context --- task results display stage and project; messages show author and snippet
      • Matched text is highlighted in yellow within each result
      • Limit of 30 results shown --- refine the query if needed

      What Happens When You Click a Result

      • Project → switches to that project's main view
      • Task → switches to the task's project and opens the task modal
      • Message → switches to the project and opens the message thread
      • Addendum → switches to the project and scrolls to the addendum log
      • Team member → switches to assignee grouping filtered to that person's tasks

      💡 Tips: Search is case-insensitive and uses substring matching. Searching "RFI" finds every task or message mentioning RFIs across all projects. Searching a person's name finds every task they're on (Lead or Support) across the whole portfolio.

      📁 Projects --- Creating & Managing

      What is a Project?

      A project represents a single bid pursuit with its own bid start date, bid due date, owner, architect, and tasks. Each project has its own task board, milestones, addendums, and team assignments.

      Creating a Project

      Click + New Project in the sidebar. Required fields: project name, bid start date, bid due date. Recommended fields: project type (commercial/healthcare/data center/etc.), delivery method, owner, architect, square footage, location, lead estimator.

      You can also paste a OneDrive URL pointing to the project's shared folder. This appears as a clickable link in the project header for fast access to drawings and specs.

      Editing a Project

      Click ✎ Edit in the project header to change any field. Project images can be uploaded for visual identification in the sidebar list.

      Bid Date Revisions

      When an addendum extends or shortens the bid, use Revise Bid Date instead of just editing the due date. This preserves the original bid date for historical reference and offers to:

      • Re-anchor all task due dates to the new bid start (preserving day offsets)
      • Refresh auto-dated milestones (if default offsets are enabled)
      • Track the revision in the Bid Countdown card

      You can reset back to the original bid dates at any time using Reset Bid Revision.

      Archiving (Past Pursuits)

      Mark a project as Won, Lost, or No-Bid using the project status dropdown. Closed projects move to the Past Pursuits collapsible section at the bottom of the sidebar --- out of the way but still accessible for reference.

      💡 Original vs. Revised dates: The system tracks both the original bid dates and revised dates. The Bid Countdown card shows whichever set is currently active. This matters for proposal narratives that reference original schedule commitments.

      ✅ Tasks --- The Core Unit of Work

      What is a Task?

      A task is a single action that needs to happen during a bid: "review architectural drawings," "send subcontractor invitations for Division 09," "review bid leveling spreadsheet," and so on. Each task has an assignee, a stage, a category, a status, a priority, and (usually) a due date.

      Adding a Task

      Click + Add Task at the top of any view. Fill in the title (what needs to happen), pick a stage and category, set priority, and either pick a due date directly OR enter a Day Offset (number of days after Bid Start) to auto-calculate the due date.

      Day Offsets --- The SBG Convention

      Every task can have a Day Offset --- a number representing how many days after the Bid Start Date this task should be done. Examples:

      • D+0 = on the bid start day
      • D+5 = 5 days after bid start
      • D+12 = 12 days after bid start

      Why this matters: when you change the bid start date (or use Revise Bid Date), all tasks with offsets automatically recalculate their due dates. You don't manually shift 30 due dates --- they shift together.

      Statuses

      • Not Started (gray) --- task exists but no work has begun
      • In Progress (navy) --- actively being worked
      • Blocked / RFI (red) --- can't proceed; waiting on info or decision
      • Pending (purple) --- waiting on someone external (sub, owner, architect)
      • Complete (green) --- done and ready for sign-off

      Priority

      High / Medium / Low. Affects the colored bar on the left edge of each card and the sort order within columns. High-priority tasks always sort to the top.

      Editing Quickly

      Three quick-edit shortcuts that don't require opening the full task modal:

      • Drag & drop a card between columns to change status (Board view)
      • Status dropdown on each card --- change status without opening the modal
      • Sortable columns in Table view --- click any header to sort by that field

      Checklists / Sub-items --- Subdivide a Task

      Some tasks have several distinct steps but don't deserve full task records of their own --- checking submission specs, gathering 5 sub bids, going through a pre-bid walkthrough list. Use the task's ✓ Checklist / Sub-items section to break the task into a checklist without cluttering your board.

      • Where to find it --- Inside any task modal, scroll past Notes to the ✓ Checklist / Sub-items collapsible section. Click to expand.
      • Adding items --- Type a step, press Enter (or click + Add). Repeat for each step.
      • Reordering --- Use the ▲ ▼ arrows on each item to move it up or down.
      • Editing text --- Click any item's text and type to edit. Changes save when you save the task.
      • Removing items --- Click ✕ next to any item, or use "Clear All" to remove everything at once.
      • Bulk check/uncheck --- Use "Check All" or "Uncheck All" to toggle every item at once.
      Visual Progress
      • Inside the modal --- A progress bar at the top of the section shows X/Y completion as a colored bar. The bar turns green when all items are checked.
      • On the board card --- A small chip shows "☐ 3/8" with a thin progress bar. When all items are complete, it turns green and shows "✓ 8/8".
      • In the table view --- A compact chip next to the task title shows "☐ 3/8" or "✓ 8/8".
      • When all items are checked --- The section header flashes briefly, suggesting (but not forcing) you to mark the parent task Complete.
      Audit Trail

      Each checked item logs who checked it and when. Hover the green checkmark badge next to a checked item to see the audit info. Useful for inspections and quality reviews where you need to know who signed off on each step.

      💡 When to use checklists vs. separate tasks: Checklists are best for steps that are part of a single accountability --- one person owns the work, and the steps are sequential or related. If different people own different steps, or steps have different due dates, use separate tasks instead.

      Recent Activity Log --- Per-Project Audit Trail

      Every project has a chronological audit log of changes made to it. Find it by expanding the 📜 Recent Activity section near the top of the project view.

      • What gets logged --- task creation, edits (with field-level diffs), status changes, deletions, project edits (name/dates/owner), milestone changes, addendum log entries, and bulk operations. Acknowledgments and sign-offs are also captured.
      • What does NOT get logged --- view changes, filter selections, modal opens without saves, search queries. The log captures meaningful changes, not navigation noise.
      • Per-event display --- each entry shows the actor (who), timestamp (relative like "2h ago" with absolute on hover), an icon for the event type, and a short summary. Edit events expand to show field-level diffs (old value crossed out → new value highlighted green).
      • Filtering --- narrow by event type (Tasks Only, Status Changes, Sign-offs, Rejections, Project Edits, Milestones, Bulk Ops) or by user.
      • Capacity --- each project keeps the most recent 500 events. Older entries auto-trim as new ones come in.
      • Display limit --- the dropdown shows up to 50 events at a time. If filters yield more matches, the count is shown so you can refine further.
      • Clear Log --- wipe the log for the current project (with confirmation). Doesn't affect tasks, milestones, or other data --- only the audit trail. Useful for sanitizing a project before exporting/sharing.

      💡 Use the actor filter to investigate. "Why did this task's due date move?" --- filter by the user who edited it, find the entry, see the from/to diff. Or sort by time to reconstruct what happened in the last hour.

      Bulk Editing --- Edit Many Tasks at Once

      To make the same change across multiple tasks (reassign, change status, shift due dates, etc.), use bulk editing instead of opening each task individually.

      • Enter selection mode --- click ☐ Select Multiple in the toolbar above the board/table. Checkboxes appear on every card and row.
      • Pick tasks --- click any card or row to toggle its selection. Selected items get a gold border and highlight. Use the master checkbox in the table header (or "Select All Visible" in the action bar) to grab everything currently filtered.
      • Floating action bar --- when you've selected at least one task, a navy bar appears at the bottom of the screen with all bulk operations: Status, Assignee, Support, Priority, Due Date, Stage, BIC, Source, and Delete.
      • Each operation opens a confirmation dialog showing exactly which tasks will be affected and what change you're about to make. Click "Apply to N Tasks" to commit.
      Bulk Operations Available
      • Status --- set all selected to Not Started / In Progress / Blocked / Pending / Complete
      • Assignee --- reassign the Lead Owner. Acknowledgment is reset so the new assignee gets prompted.
      • Support --- add a person as support across many tasks, remove a person from support, or clear all support members
      • Priority --- set Low / Medium / High
      • Due Date --- set all to a specific date, shift by N days (positive or negative), or clear all dates
      • Stage --- move all selected to a different bid stage
      • Ball in Court --- change BIC owner
      • Source --- change source (Bluebeam, BuildingConnected, etc.)
      • Delete --- bulk delete with confirmation showing the first 5 task titles

      💡 Bulk editing respects active filters. If you've filtered the board to only show "Bluebeam-sourced tasks" with the Group by Source feature, then "Select All Visible" only selects those. Combine filtering and bulk editing for surgical changes.

      💡 Date shifting is bid-aware. When you shift due dates by N days, tasks that had a day-offset relative to the bid start date have their offset updated too. So if you push everything 3 days later because the bid date moved, both the absolute date and the relative offset stay consistent.

      Recurring Tasks

      For tasks that repeat --- weekly status updates, bi-weekly bid reviews, monthly reports --- set up a recurrence pattern in the task modal's 🔁 Recurrence section. When you mark the task Done, the next instance is automatically created with a new due date.

      • Patterns supported --- Daily, Weekly (with optional day-of-week selection), Bi-weekly, Monthly (with optional day-of-month), and Custom (every N days).
      • Day-of-week picker for weekly tasks --- click Mon/Wed/Fri to make a task recur on those specific days. Leave all unselected to repeat on the same day each week.
      • End conditions --- set an end date (e.g., "until bid due date") OR a max occurrence count (e.g., "12 weekly meetings"). Leave both blank to recur indefinitely.
      • How it works --- When you complete a recurring task, a brief confirmation tells you the next instance was created with its computed due date. The new instance inherits everything (assignee, support, notes, source) except status and acknowledgment state --- those reset for the new cycle.
      • Series tracking --- instances are linked together. Open any instance to see "Part of a recurring series · Instance #3 · 12 total · 2 completed."
      • Visual indicator --- recurring tasks show a green 🔁 badge on board cards and table rows so they stand out as routine work.
      • Editing the pattern --- change the pattern on any instance and it applies forward (existing instances stay as-is, future instances use the new pattern).
      • Stopping a series --- change the pattern to "None --- one-time task" on any instance, save, then mark Done. No further instances will be created.

      Lead Owner & Support Members

      Each task has one Lead Owner (the single point of accountability) and one or more Support Members (people helping the Lead). Single-point Lead accountability is intentional --- it prevents the "everyone is responsible, nobody is" pattern that diffuses ownership.

      • Adding multiple supports --- In the task modal's Support Members field, type a name and press Enter (or click + Add). The name appears as a chip with their avatar. Repeat to add more. Click the ✕ on any chip to remove that person.
      • Where supports show up --- Board cards display stacked support avatars (up to 3 inline + "+N more" badge). Look Ahead view, Today view, and the team grouping all include support members in their filters.
      • Notifications --- When a task is edited, completed, or rejected, every Support member gets notified via task messages, alongside the Lead Owner.
      • Acknowledgment authority --- Any Support member can acknowledge a completion rejection (in addition to the Lead Owner). The Lead Estimator's project-level sign-off authority is unchanged --- Support membership doesn't grant that.
      • Reassigning --- The Reassign Tasks function on a team member's row swaps their name across both the primary support and any additional support memberships, in one operation.

      Saving to Master Template

      Found a task that should be standard for every project? Open the task → click ⭐ Save to Master Template. The template fields (title, stage, category, source, priority, offset) get added to the master list. Per-project fields (assignee, status, due date, notes) are NOT copied.

      Emailing About a Task

      Open any saved task → click ✉ Email Task. The dialog pre-fills a draft with the project name, task title, due date, assignee, status, and notes --- then opens in your default email client (Outlook for most SBG users) so you can review and send.

      • Quick-pick recipients --- buttons for the assignee, support, lead estimator, and rest of the roster. Add team email addresses in Manage Team first; otherwise the button drops in a placeholder you can replace.
      • Body is editable --- the auto-generated body is a starting point, not final copy. Trim or expand before sending.
      • Long emails --- most clients accept up to roughly 2,000 characters in a mailto link. For longer messages, click 📋 Copy Body to copy the full text and paste it manually after the draft opens.

      Adding a Task to Your Calendar

      Open any saved task → click 📅 Add to Calendar. The dialog pre-fills an event with the task title, defaults the date to the task due date, and includes the project context in the description. Two ways to send it to your calendar:

      • 🌐 Open in Outlook Web (recommended for SBG) --- opens Outlook on the web in a new tab with all fields pre-filled. One click, no download. Works as long as you're signed into Outlook on the web in your browser. If you're not signed in, you'll see the Microsoft 365 login page first; after signing in, the event form appears.
      • 📥 Download .ics --- generates a standard calendar file that opens in any calendar app. Use this if your primary calendar is Outlook desktop only, or if your browser blocks the Outlook Web popup. Double-click the downloaded file to open it in Outlook.
      • Personal block vs. meeting invite --- leave Attendees blank to add a personal time block. Add attendees and Outlook will offer to send the event as a meeting invite to those people.
      • All-day or timed --- pick a duration from 15 minutes up to several hours, or choose "All-day event" for full-day blocks.
      • Reminder --- defaults to 1 day before. Choose anything from 15 minutes to a week before. (Reminder is honored on .ics downloads; Outlook Web uses your default reminder setting and lets you adjust before saving.)
      • Show as Busy/Free --- checkbox controls whether the event blocks your availability. Uncheck for tentative or informational events. (Outlook Web honors the busy/free choice when set on the event after opening.)
      • If popup blocker stops the Outlook Web tab --- the dialog will offer to copy the link to your clipboard so you can paste it into a new tab manually.

      Adding Project Milestones to Your Calendar

      Project milestones (Trade Partner Bids Due, RFIs Due, Bid Submission, etc.) can also be sent to your calendar. Two ways to do it:

      • Single milestone --- Open Key Milestones from the project page → click the green 📅 button on any milestone row with a date set. Same dialog as task calendar invites, pre-filled with the milestone's date, time, and project context. Use Outlook Web or .ics download as before.
      • Bulk export --- all milestones at once --- From Key Milestones, click 📅 Add All to Calendar at the bottom. This generates a single .ics file containing every milestone with a date set (skipping any that are missing dates), with a one-day reminder on each. Open the file in Outlook to add the entire bid timeline to your calendar in one operation.

      Tip: Milestone calendar events default to 5:00 PM if no time is set, with a 30-minute duration and 1-day reminder. The description includes project context (owner, architect, bid due date), lead time relative to the bid, and your project notes. Outlook tags them with category "SBG Bid Milestone" so you can filter your calendar later.

      👥 Team & Assignments

      Team Roster

      Sidebar → Manage Team. The master team roster lists everyone available for assignment across all projects. Each member has a name, title (Estimator I/II/Senior, Precon Director, Operations, etc.), and optional photo.

      Lead Owner vs. Support

      • Assignee (Lead Owner) --- the person responsible for getting the task done. Their name appears as the primary avatar on the card. They get acknowledgment alerts.

      Ball in Court (BIC)

      • Reassign an entire stage from a departing team member to a replacement
      • Setting Support across all tasks in a stage for cross-training

      Reassign All Tasks (per person)

      Team Members modal → ↻ Reassign Tasks button on a member's row. Moves every task currently assigned to one person over to another, with options for scope (current project or all projects), roles (Lead / Support / both), and whether to notify the new owner via acknowledgment reset.

      Use cases: vacation handoff, departure handoff, mid-project reassignment.

      ✓ Acknowledgment & Sign-Off Lifecycle

      Why Acknowledgment Exists

      When you assign a task to someone, how do you know they actually saw it? Acknowledgment is the explicit "I see this and I own it" handshake. It prevents tasks from quietly sitting unowned because the assignee didn't realize they had it.

      The Flow

      1. Lead Estimator (or anyone) assigns a task to Sarah
      2. Task appears in Sarah's My Alerts → Unacknowledged tier with a NEW badge
      3. Sarah clicks ✓ Acknowledge on the card or in the task modal --- confirms she sees it
      4. The NEW badge disappears; the task is now part of her normal workload

      Lead Override

      Sometimes the assignee can't acknowledge in time (out of office, on a flight, etc.) but the task still needs to move. The Lead Estimator can ⚡ Override to acknowledge on the assignee's behalf. This is logged with an audit badge so it's clear the acknowledgment came via override, not from the actual assignee.

      Completion Sign-Off

      When a task is marked Complete, the Lead Estimator can sign off on it (✓ Signed off by [Lead] · [date]). This creates a documented "I've reviewed this and confirm it's done" stamp. Useful for QA before bid day.

      Completion Rejection

      If a task was marked Complete but the work wasn't actually finished, the Lead can reject the completion with a reason. The task gets a red REJECTED badge with the reason chip. The original assignee then has to acknowledge the rejection and redo the work.

      💡 Audit trail: Every acknowledgment, override, sign-off, and rejection is timestamped with who did it. This becomes your bid-day defense if anyone asks "why did this task slip?"

      🔔 Alerts & Messages

      Alert Tiers

      The Alerts section at the top of every project (and in the Today view) groups tasks by urgency:

      • Overdue --- past due date, not yet complete (red, top priority)
      • Imminent --- due today or tomorrow (orange)
      • Due Soon --- due in exactly 2 days (yellow)
      • Unacknowledged --- newly assigned, not yet acknowledged by the assignee

      The Alerts section splits into two:

      • Project Alerts --- everything in this project that's in alert state, regardless of who owns it. Visible to everyone, useful for Lead Estimators monitoring the whole bid.
      • My Alerts --- only YOUR tasks (matches the Current User in Settings). Your personal action list.

      Task Message Threads

      Every task has its own message thread (visible in the task modal). Use it for back-and-forth between the Lead, the assignee, and any support members. Messages can include @mentions to notify other team members.

      @Mentions

      📚 Master Lists --- System-Wide Customization

      What Are Master Lists?

      Master lists are the configurable values that show up in dropdowns across every project. They live in the sidebar's Templates & Settings section. Editing a master list affects every project immediately.

      Stages

      The 9 default bid workflow stages: Project Setup, Special Project Tasks, Document Review, Trade Solicitation, Estimating, Scope Leveling, Bid Day Prep, Bid Day, Post-Bid. You can edit stage names, icons, descriptions, reorder them, and add custom stages. Stages are how tasks are grouped on the Board view.

      Master Task Template

      The reusable workflow blueprint. When creating a new project, you can load this template to instantly populate dozens of standard tasks with calibrated day offsets. Tasks can be added directly to the template OR promoted from a real project task using ⭐ Save to Master Template.

      Ball in Court List

      The list of BIC categories (Lead, Support, Owner, Architect, Subcontractor, etc.) with their colors. Customize for your workflow.

      CSI / Division List

      The CSI MasterFormat divisions used as task categories (00 Procurement, 01 General Requirements, 03 Concrete, 09 Drywall, etc.). Edit codes and names as needed for SBG's conventions.

      Source List

      Where a task came from: Bluebeam markup, Building Connected invitation, email, etc. Each source has an emoji icon (auto-suggested by name) and optional uploaded logo image. Source badges appear on board cards and table rows.

      Milestone Types

      The categories of key bid milestones (Trade Partner Bids Due, RFIs Due from Subs, Pre-bid Walkthrough, Bid Leveling Review, Executive Review, etc.). Each can have a default day offset for auto-computation when projects opt in.

      🎯 Key Milestones --- Critical Dates

      Milestones vs. Tasks

      Milestones are time-anchored events (Pre-bid Walkthrough at 10am on April 21, RFIs Due from Trade Partners on April 19, Executive Review on April 24). They're separate from tasks because they typically have a specific date AND time, are referenced by date in proposals, and need to be visible at-a-glance without drilling into task lists.

      The Default Offset System

      Each milestone TYPE in the master list (Trade Partner Bids Due, RFIs Due, etc.) can have a Days Before Bid default. Examples: Trade Partner Bids Due = 1 day before bid; RFIs Due = 7 days before bid; Bid Leveling = 0 (on bid day); Executive Review = 2 days before bid.

      Per-Project Opt-In

      Projects don't automatically use default offsets --- they opt-in. Open the Key Milestones modal → toggle Use default offsets from bid due date. The system asks whether to apply defaults now (overwriting existing dates) or just enable for future bid date changes.

      When opted in, milestone dates auto-compute as (bid due - offset days). When the bid due date changes (via Edit Project or Revise Bid Date), auto-dated milestones recalculate automatically.

      Manual Override Stays Manual

      If you manually edit a milestone date, that milestone is "yours" --- it won't get overwritten when the bid date changes. The visual chip switches from 🔗 Auto (navy) to ↺ Default available (gold). To re-arm it with auto-computation, click Re-Apply Defaults Now at the top of the milestone editor.

      Visual Indicators

      • 🔗 Nd (navy chip) --- auto-dated, will refresh on bid date changes
      • ↺ Nd (gold chip) --- default available but not currently applied
      • Nd (gray chip) --- default exists but project hasn't opted in
      📜 Addendums Log

      Why Track Addendums?

      Architects and owners issue addendums during the bid period to clarify scope, add drawings, change specifications, or extend the bid date. Tracking them in the project gives you a single audit log of every change --- useful for proposal narratives, scope verification, and explaining "why our bid changed mid-pursuit."

      Adding an Addendum

      Project header → 📜 Addendums button (or scroll to the Addendums section). Click + Add Addendum. Fill in:

      • Number (auto-numbered if blank: #1, #2, etc.)
      • Subject (e.g., "Architectural revisions to lobby")
      • Date Issued / Date Received
      • Issuer (architect, owner, GC, etc.)
      • Description of what changed
      • Affected Documents (drawings, specs, exhibits)
      • Bid Impact --- None, Extends, Shortens, or TBD. If extends/shortens, enter the new bid date.
      • Attachment (filename + URL link)

      Bid Date Extension Flow

      If you set Bid Impact = Extends or Shortens AND enter a new bid date, the system asks if you want to open Revise Bid Date with that new date pre-filled. This connects the addendum log to the bid revision flow so the schedule gets properly updated.

      Status Tracking

      Each addendum has a status: Received → Reviewed → Distributed → Acknowledged. Quick-update via the dropdown on each addendum row.

      📦 Scope Packages

      What Are Scope Packages?

      Scope packages organize tasks by trade or work scope independently of CSI division. Examples: "Foodservice Equipment," "Mass Timber Structure," "Architectural Precast." They're useful when a single bid scope cuts across multiple CSI divisions (Foodservice = Div 11 + Div 22 + Div 26 + electrical) or when you want to see "all bidding work for this scope" together.

      Assigning to a Scope Package

      Open any task → set Scope Package field. Each scope package has a name, icon, and color used on cards and in the Schedule view's "By Scope" grouping.

      Why Use Them?

      • Schedule view → Group by Scope shows your entire bid by trade scope
      • Quickly check coverage: do you have bids in for every active scope?
      • Track progress: which scopes are complete vs. still gathering bids?

      📖 Glossary --- SBG & Industry Terms
      BIC (Ball in Court)
      Whose action is currently needed to move a task forward. Different from assignee --- assignee owns the task overall, BIC says who has the ball right now.
      Bid Start / Bid Due
      The two anchor dates for a bid pursuit. Bid Start = when the bid period opens (you receive RFP). Bid Due = when the proposal must be submitted.
      BuildingConnected (BC)
      Autodesk's bid invitation platform. SBG uses it for sub solicitation and ITB distribution.
      CSI / MasterFormat
      Construction Specifications Institute's standard taxonomy for organizing construction work by division (Div 03 Concrete, Div 09 Finishes, etc.).
      D+N
      Day offset notation. D+5 = 5 days after Bid Start. Used to schedule tasks relative to the bid timeline so they auto-shift when bid dates change.
      Day Offset
      Number of days after Bid Start when a task is due. The system stores this AND a calculated due date --- change either and the other updates.
      ITB
      Invitation to Bid. The package SBG sends to subcontractors inviting them to bid on a scope.
      JOC
      Job Order Contract. A contract delivery method where SBG holds an indefinite-quantity master agreement and gets task orders against it (Dallas County JOC, Parkland Health MATOC).
      Lead Estimator
      The senior estimator with overall responsibility for a project's bid. Has Override permission for acknowledgment and Sign-Off authority on completed tasks.
      Lead Owner (Assignee)
      The person responsible for a specific task. Distinct from the Lead Estimator (project-wide role) --- a task's Lead Owner could be any team member.
      Master Template
      The reusable workflow blueprint of standard tasks. Loaded into new projects to skip 30+ minutes of manual setup.
      OCIP
      Owner-Controlled Insurance Program. Owner provides project insurance coverage instead of each contractor providing their own. Affects subcontract terms.
      RFI
      Request for Information. A formal question to the architect/owner during bid period (and during construction). Tracked as tasks and as milestones (RFIs Due dates).
      RFP / RFQ
      Request for Proposal / Request for Qualifications. The owner's solicitation document.
      ROM
      Rough Order of Magnitude. An early-stage budget estimate based on limited information.
      Scope Package
      A grouping of related work for bidding purposes (e.g., Foodservice Equipment, Mass Timber). Independent of CSI division.
      Sign-Off
      Lead Estimator's confirmation that a Complete task has been reviewed and is genuinely done. Creates an audit timestamp.
      TABS / TDLR
      Texas Department of Licensing and Regulation's Texas A&M Bid System (or similar agency systems) where public projects are posted.
      Trade Partner
      SBG's term for a subcontractor. Used in milestones (Trade Partner Bids Due) and outreach.
      ❓ FAQ & Common Confusions

      I can't see my tasks in My Alerts.

      Most likely cause: Current User isn't set. Open Settings → Current User and enter your name exactly as it appears in the Team Roster. The system filters My Alerts by exact name match.

      I changed the bid due date and now all my milestones moved.

      That's expected if you opted in to default offsets. Auto-dated milestones recompute when bid date changes. To prevent this for specific milestones, manually edit those dates in the Key Milestones modal --- they'll switch to "manual" and won't auto-refresh.

      Why doesn't the Calendar view show my task?

      Tasks need a due date to appear on the Calendar. Tasks with only a day offset will have a calculated due date IF the project has a Bid Start Date set. Set the Bid Start in Edit Project.

      I dragged a task to Complete but it shows "Awaiting Sign-Off."

      Working as designed. Lead Estimators sign off on completed tasks to confirm review. The Lead opens the task and clicks Sign Off. The task gets a "✓ Signed off by [Lead]" footer once approved.

      I deleted a stage but tasks still reference it.

      Stage deletion prompts you to reassign tasks to a different stage first. If somehow tasks got orphaned (e.g., import from older data), they appear in an "Other" group on the Schedule view and a "(legacy)" option in the Stage dropdown.

      How do I print a clean copy for a meeting?

      Use Browser Print (Ctrl+P / Cmd+P) on any view. The system has print-specific styling that hides the sidebar and modal backdrops. Schedule view prints landscape with the time axis preserved.

      The Schedule view says "No tasks to schedule."

      Tasks need a due date to render on the Schedule. If you have many tasks with no due date, they're filtered out. The title bar tells you "X of Y tasks" so you can see how many are filtered. To fix, add due dates (or day offsets --- those compute due dates automatically when Bid Start is set).

      Can multiple people use this on the same device?

      Yes --- each user updates the Current User in Settings before working. Acknowledgments, sign-offs, and messages are attributed to whoever the Current User was at the time of the action. (The audit trail records WHO did each action and WHEN.)

      I made a mistake on a bulk operation. Can I undo?

      There's no undo button. Best practice: Export JSON before any bulk operation (Bulk Assign, Reassign Tasks, Apply Default Offsets, Reset Bid Revision). If something goes wrong, Import the backup to restore.

      My task disappeared!

      Check three things: (1) the project's filter --- you might be on a single stage tab while the task is in a different stage, (2) the assignee filter --- same issue with by-Assignee grouping, (3) the search/scope filters at the top of the section. Switch grouping to "All Stages" or "All Team" to see everything.
